# 课题组科研成果文件管理软件初始化计划

**计划日期**: 2026-06-23  
**计划文件**: `docs/plans/init-20260623.md`  
**当前状态**: 仅制定计划，暂不开始实现  
**需求来源**: 用户提供的《课题组科研成果文件管理软件需求框架》  

## 目标

将当前项目从“文献管理 CLI 原型”升级规划为“课题组科研成果与相关文件管理系统”。系统不仅管理论文文献，还需要覆盖文章、专利、软件著作权、会议成果、项目/基金材料、数据代码与支撑文件，并支持角色权限、审核、附件、统计、导出和操作日志。

本计划只定义后续实施路径，不直接修改源码、数据模型或功能逻辑。

## 范围理解

### 核心定位

系统用于课题组科研成果与相关文件的统一管理，服务于：

- 成果沉淀：集中保存文章、专利、软著、会议成果、项目材料、数据代码等。
- 责任明确：记录每项成果负责人、参与人、所属项目和审核状态。
- 证明完整：保存全文、证书、录用通知、检索证明、会议证明、申报材料等附件。
- 管理汇总：支持 PI / 老板查看全组成果、统计面板、导出年度总结和项目申报材料。
- 成员协作：成员按权限维护自己的成果，并提交给管理员或 PI 审核。
- 合规可追溯：关键操作保留日志，删除严格受控。

### 主要对象

- 成果记录：文章、专利、软件著作权、会议成果、项目/基金材料、数据/代码/支撑文件。
- 用户与角色：PI、管理员/科研秘书/助理、普通成员、只读成员/访客。
- 权限：查看、新增、编辑、删除、归档、附件上传、附件下载、导出、审核、权限管理、日志查看。
- 附件：PDF、Word、图片、证书、通知、证明、数据包、代码包、压缩文件等。
- 审核流：草稿、待审核、退回修改、审核通过、归档、作废/删除申请。
- 统计导出：按年份、类型、成员、项目、状态、期刊分区、专利状态等统计。

## 关键设计原则

- 先统一领域模型，再扩展界面：不要直接从 UI 开始堆功能。
- 权限和审核从第一版就纳入模型，避免后期补丁式改造。
- 附件只保存元数据和受控路径，不把大文件或隐私文件提交到 Git。
- 删除默认软删除或作废，真实删除需 PI/管理员确认并保留日志。
- 字段设计分“通用字段 + 类型专属字段”，避免每类成果重复一套完全割裂的结构。
- 统计导出优先服务年度总结、项目申报、职称材料和结题材料。
- 先实现单机/本地原型，再评估 Web 多用户和数据库。

## 建议技术路线

### 阶段 0：需求固化与边界确认

目标：把当前需求框架整理成可实现的产品说明。

任务：

- 整理成果类型清单。
- 整理角色与权限矩阵。
- 整理每类成果字段。
- 定义审核状态和归档状态。
- 定义附件类型和命名规范。
- 明确当前版本是否只做本地原型，还是直接做 Web 多用户系统。

产出：

- `docs/product/REQUIREMENTS.md`
- `docs/product/PERMISSION_MATRIX.md`
- `docs/product/ENTITY_FIELDS.md`

暂定结论：当前建议先做本地可验证原型，再迁移到 Web/数据库。

### 阶段 1：统一领域模型重构

目标：把现有“文献记录”扩展为“科研成果记录”。

建议模型：

- `ResearchOutput`
  - 通用字段：编号、标题、成果类型、状态、负责人、参与人、所属项目、年份、关键词、摘要/说明、备注、创建时间、更新时间。
  - 类型专属字段：文章字段、专利字段、软著字段、会议字段、项目材料字段、数据代码字段。
- `Member`
  - 姓名、角色、邮箱/工号、课题组状态、备注。
- `Project`
  - 项目名称、项目类型、编号、负责人、起止时间、资助来源。
- `Attachment`
  - 文件名、附件类型、路径、所属成果、上传人、上传时间、校验信息、可下载权限。
- `AuditLog`
  - 操作人、操作类型、对象类型、对象 ID、变更前后摘要、时间。
- `ReviewRecord`
  - 提交人、审核人、审核状态、审核意见、时间。

实施要点：

- 保留当前 `LiteratureRecord` 的价值，但不要让它成为所有成果的唯一模型。
- 可以新增 `ResearchOutput`，并将文章作为一种成果类型。
- 现有 CLI 可逐步从 `literature` 命名迁移到 `outputs` 或 `research_outputs`。

### 阶段 2：权限模型与审核流设计

目标：建立可测试的权限判断函数，而不是只在文档里写权限。

建议权限枚举：

- `view`
- `create`
- `edit`
- `delete`
- `archive`
- `upload_attachment`
- `download_attachment`
- `export`
- `review`
- `manage_permissions`
- `view_audit_log`

建议角色：

- `pi`
- `admin`
- `member`
- `readonly`

建议审核状态：

```text
draft -> submitted -> returned -> approved -> archived
```

删除建议：

```text
active -> delete_requested -> voided / deleted_by_admin
```

实施要点：

- 普通成员不能直接删除。
- 普通成员默认只能编辑自己负责或参与的成果。
- PI 拥有全局查看、审核、导出和权限管理能力。
- 管理员可协助维护成果与附件，但权限管理可按配置控制。
- 只读成员默认不能编辑、删除、上传。

### 阶段 3：存储层升级

目标：从单一 `library.json` 过渡到更适合多实体的本地存储结构。

本地原型建议结构：

```text
data/local/
├── outputs.json
├── members.json
├── projects.json
├── attachments.json
├── audit_logs.json
└── reviews.json
```

后续 Web 版可迁移到 SQLite / PostgreSQL。

实施要点：

- JSON 原型必须保留 schema version。
- 每个实体使用稳定 ID。
- 存储层要有导入导出函数。
- 写入前进行基础校验。
- 对损坏 JSON 保持当前的友好错误处理策略。

### 阶段 4：CLI 原型扩展

目标：先用 CLI 验证核心业务闭环。

建议命令：

```bash
# 成果
outputs add
outputs list
outputs show
outputs update-status
outputs submit
outputs approve
outputs return
outputs archive
outputs request-delete

# 成员
members add
members list
members show

# 项目
projects add
projects list
projects show

# 附件
attachments add
attachments list
attachments verify

# 统计与导出
stats summary
export csv
export annual-report
```

第一批 MVP 命令建议只做：

- `outputs add`
- `outputs list`
- `outputs show`
- `outputs submit`
- `outputs approve`
- `stats summary`
- `export csv`

### 阶段 5：文章模块字段落地

目标：优先把用户提供最详细的“文章模块”落地。

文章类型：

- 中文论文
- 英文论文
- 研究论文
- 综述
- Meta 分析
- 病例报告
- 方法学论文
- Letter / Editorial / Commentary

文章字段：

- 文章标题
- 中文标题
- 英文标题
- 文章类型
- 期刊名称
- ISSN
- DOI
- PMID
- 发表年份
- 卷、期、页码
- 影响因子
- JCR 分区
- 中科院分区
- 投稿状态
- 投稿日期
- 接收日期
- 在线发表日期
- 正式发表日期
- 第一作者
- 共同第一作者
- 通讯作者
- 共同通讯作者
- 其他作者
- 所属成员
- 所属项目 / 基金
- 关键词
- 摘要
- 备注

文章状态：

```text
选题中 -> 写作中 -> 内部修改 -> 投稿中 -> 返修中 -> 接收 -> 在线发表 -> 正式发表 -> 归档
```

实施要点：

- 投稿状态可作为文章专属字段，不等同于系统审核状态。
- 作者字段应支持多个成员和外部作者。
- 影响因子和分区可能每年变化，应记录年份或备注。

### 阶段 6：其他成果模块逐步落地

优先级建议：

1. 专利
2. 软件著作权
3. 会议成果
4. 项目 / 基金材料
5. 数据、代码与支撑文件

每类成果都按以下模板整理：

- 类型枚举
- 必填字段
- 可选字段
- 状态流
- 附件类型
- 统计维度
- 导出字段

### 阶段 7：附件管理与文件规范

目标：建立真实文件管理边界。

建议附件类型：

- 论文全文 PDF
- 投稿/接收/发表证明
- 专利受理通知书
- 专利授权证书
- 软著证书
- 会议邀请函
- 会议日程
- 壁报文件
- 发言 PPT
- 项目申报书
- 合同/任务书
- 结题材料
- 数据说明
- 代码压缩包

建议附件目录：

```text
data/local/files/
├── articles/
├── patents/
├── software_copyrights/
├── conferences/
├── projects/
└── datasets/
```

命名建议：

```text
年份_成果类型_负责人_简短标题_附件类型.ext
```

注意：真实附件目录必须被 `.gitignore` 忽略。

### 阶段 8：统计、筛选与导出

目标：服务 PI 和管理员的汇总需求。

筛选维度：

- 年份
- 成果类型
- 成员
- 负责人
- 项目 / 基金
- 审核状态
- 归档状态
- 期刊分区
- 专利状态
- 会议类型

导出形式：

- CSV
- Excel（后续）
- Markdown 年度总结
- 项目申报成果清单
- 职称材料清单
- 结题材料清单

MVP 先支持 CSV 和 Markdown。

### 阶段 9：Web 化评估

当前 CLI 原型适合验证业务模型，但多用户权限和附件下载最终更适合 Web 系统。

候选方向：

- FastAPI + SQLite/PostgreSQL + 简单前端
- Django Admin 快速后台
- Streamlit 内部工具原型

初步建议：

- 如果目标是快速内部使用：Django Admin 或 Streamlit。
- 如果目标是长期可维护 Web 系统：FastAPI + 前端 + PostgreSQL。
- 在领域模型稳定前，不建议立即投入复杂前端。

## 初始 MVP 建议

第一版不要覆盖所有功能，建议先做“成果记录 + 文章模块 + 权限判断 + 审核流 + CSV 导出”的闭环。

### MVP 必做

- 成果统一模型 `ResearchOutput`
- 成果类型枚举
- 文章专属字段
- 成员模型
- 项目模型
- 角色与权限判断函数
- 审核状态流
- JSON 本地存储
- CLI：新增、列表、查看、提交、审核
- CSV 导出
- 测试覆盖核心权限和状态流

### MVP 暂不做

- Web UI
- 多人同时编辑
- 真正的登录系统
- 大文件上传服务
- Excel 高级格式
- 自动抓取 DOI / PubMed / 专利数据库
- 全文检索

## 风险与待确认问题

### 产品问题

- 是否一定需要多人在线使用？
- 老板/PI 是否需要手机或网页查看？
- 成员是否需要登录账号，还是先由管理员代维护？
- 附件是否需要在线预览？
- 是否需要和 Zotero、EndNote、PubMed、专利数据库同步？

### 数据问题

- 当前课题组已有成果是否有 Excel 表？
- 成员名单和项目清单是否已有固定格式？
- 附件目前分散在哪里？
- 是否允许把真实附件统一复制到系统目录？
- 是否需要导入历史成果？

### 权限问题

- 普通成员是否能看到全组成果标题？
- 是否允许成员下载其他人成果附件？
- 管理员是否能删除，还是必须 PI 删除？
- 审核通过后普通成员是否还能修改？

### 技术问题

- 最终部署在个人电脑、课题组服务器，还是 NAS？
- 是否有校内服务器或内网访问要求？
- 是否需要备份策略？
- 是否需要离线使用？

## 建议下一步

4. 在下一轮实现前，先用计划模式设计 `ResearchOutput` 数据模型和 CLI 命令迁移方案。
5. 获得确认后，再开始修改源码，不在本计划阶段直接实现。
