# 项目路线图

**创建日期**: 2026-06-23  
**最近更新**: 2026-06-23  
**适用范围**: 课题组科研成果与相关文件管理程序

## 目标

把本项目逐步建设为课题组内部可持续维护的科研成果管理工具：先验证本地统一领域模型、权限和审核流，再扩展附件、导入导出、数据库和 Web 化能力。

## 当前 MVP 范围

当前 MVP 已实现最小闭环：

- 使用统一 `ResearchOutput` 模型管理文章、专利、软著、会议成果、项目材料和数据代码成果类型。
- 使用 `Member`、`Project` 与角色权限模型表达负责人、参与人和项目归属。
- 使用本地 JSON 多实体文件保存成果、成员、项目、审核记录和审计日志。
- 通过 CLI 管理成员、项目与成果，并完成 `draft -> submitted -> approved` 审核闭环。
- 提供汇总统计和 CSV 导出。
- 使用标准库测试保护模型、权限、存储、CLI 和导出行为。

MVP 暂不承诺：

- 多人实时协作与登录系统。
- 真实附件上传、在线预览和版权治理。
- Excel 高级格式或复杂报表模板。
- 云同步或外部数据库。
- Web UI。

## 阶段计划

### 阶段 1：本地统一领域模型

- 固化 `ResearchOutput`、`Member`、`Project`、`ReviewRecord`、`AuditLog`。
- 补齐文章以外成果类型的专属字段。
- 为所有集合维持 `schema_version` 与稳定 ID。

### 阶段 2：审核流与权限强化

- 增加 `returned`、`archived`、`request-delete` 流程命令。
- 把角色权限矩阵扩展到附件下载、导出和日志查看。
- 为关键状态迁移补充更细粒度测试。

### 阶段 3：附件与导入导出

- 增加附件元数据实体与目录规范。
- 支持历史成果 CSV 导入与校验报告。
- 增加 Markdown 年报导出和项目申报清单导出。

### 阶段 4：存储升级与 Web 评估

- 评估 SQLite 作为本地增强版后端。
- 评估 Django Admin、FastAPI 或 Streamlit 内部原型。
- 根据真实使用反馈决定是否迁移到数据库和 Web 权限体系。

## 设计原则

- 数据安全优先：真实 PDF、证书、本地路径、私密笔记和成员隐私默认不进入 Git。
- 先小后大：优先保持本地 CLI 与 JSON 闭环稳定，再上更重的技术栈。
- 可迁移：当前模型和导出格式应为后续数据库/Web 版本提供迁移基础。
- 可测试：每次修改模型、权限、审核流或 CLI，都必须补充回归测试。

## 下一步建议

1. 用虚构或公开数据试录入 10-20 条成果，验证字段是否覆盖真实工作流。
2. 补充专利、软著、会议和项目材料的类型专属字段表。
3. 增加附件元数据和附件命名规范的 CLI 支持。
4. 在模型稳定后再决定 SQLite 或 Web 化路线。
